# Database Design

Neon Postgres (serverless). Two schemas: `neon_auth` (managed by Neon Auth / Better Auth) and `public` (application tables managed by Drizzle ORM).

Fields marked with **[E]** are application-level encrypted (AES-256-GCM) before storage. See `docs/security.md` for key architecture.

---

## Application Schema (`public`)

Managed by Drizzle ORM. Schema defined in `lib/db/schema.ts`.

### `user_profiles`

Extends Neon Auth users with app-specific data (subscription, tier). Created via webhook or on first login.

| Column | Type | Notes |
|---|---|---|
| `id` | `uuid` | PK, auto-generated |
| `auth_user_id` | `text` | Unique, FK -> neon_auth.user.id |
| `tier` | `text` | `free` or `pro`, default `free` |
| `polar_customer_id` | `text` | **[E]** Polar customer ID, nullable |
| `polar_subscription_id` | `text` | **[E]** Active Polar subscription ID, nullable |
| `subscription_status` | `text` | `active`, `canceled`, `past_due`, `trialing`, nullable |
| `billing_cycle` | `text` | `monthly` or `yearly`, nullable |
| `current_period_end` | `timestamptz` | When current billing period ends, nullable |
| `created_at` | `timestamptz` | |
| `updated_at` | `timestamptz` | |

Tier enforcement:
- `free` -- 1 project max, 40 tasks max, no org membership
- `pro` -- unlimited projects/tasks, can join/create orgs

### `api_keys`

Unified storage for LLM API keys (personal and org). Write-only -- keys are never readable after saving.

| Column | Type | Notes |
|---|---|---|
| `id` | `uuid` | PK, auto-generated |
| `owner_type` | `text` | `user` or `org` |
| `owner_id` | `text` | User ID or org ID |
| `provider` | `text` | `google`, `anthropic`, `openai` |
| `model` | `text` | Preferred model, nullable |
| `encrypted_key` | `text` | **[E]** AES-256-GCM ciphertext |
| `key_hint` | `text` | Last 4 chars for UI display (e.g. `••••a1b2`) |
| `created_at` | `timestamptz` | default now |
| `updated_at` | `timestamptz` | default now |

Unique constraint on `(owner_type, owner_id, provider)`.

Key resolution order per chat request:
1. Personal key (`owner_type='user'`, `owner_id=currentUser.id`)
2. Org shared key (`owner_type='org'`, `owner_id=activeOrgId`)
3. Server env var fallback

## Cascade Behavior

```
organization (Neon Auth)
  └── member (Neon Auth, cascade)
  └── invitation (Neon Auth, cascade)
  └── projects (via org_id -- SET NULL or application-level handling)

user (Neon Auth)
  └── user_profiles (application cascade)
  └── api_keys WHERE owner_type='user' (application cascade)
  └── member (Neon Auth cascade)
  └── project_shares (application cascade)

projects
  └── phases (cascade)
      └── modules (cascade)
          └── tasks (cascade)
  └── conversations (cascade)
  └── project_shares (cascade)
```

`node_edges` have no FK constraints (polymorphic) -- orphan cleanup is application-level.

## GDPR Compliance by Design

### Data Isolation

Each org/user has a derived encryption key (`HKDF(masterKey, scopeId)`). Data is cryptographically isolated -- even with full DB access, one org's data cannot decrypt another's.

### Right to Erasure (Art. 17)

Deleting a user triggers:

1. Delete `api_keys` rows for the user (encrypted LLM keys destroyed)
2. Delete `user_profiles` row (subscription data)
3. Delete all `project_shares` for the user
4. Delete owned personal projects (cascade removes phases, modules, tasks, conversations, edges)
5. Remove `member` rows from all orgs (Better Auth handles this)
6. Delete `neon_auth.user` row (email, name, avatar gone)

After deletion, the user's derived key scope is unreachable. Any encrypted data remaining in backups is permanently unreadable.

### Right to Access / Portability (Art. 15, 20)

Export endpoint decrypts and returns all user data as JSON:

```json
{
  "account": { "email": "...", "name": "..." },
  "organizations": [...],
  "projects": [
    {
      "title": "...", "description": "...",
      "phases": [...],
      "modules": [...],
      "tasks": [...],
      "conversations": [...]
    }
  ]
}
```

### Data Minimization

| Data | Stored where | Why |
|---|---|---|
| Email, name, avatar | `neon_auth.user` | Account identity |
| Project content | `public.*` (encrypted) | Core product functionality |
| LLM API keys | Client localStorage only | Never sent to or stored on our servers |
| Payment/billing | Polar.sh only | Never touches our DB |
| Analytics/tracking | Nowhere | Not collected |

### What an Attacker Sees in a Full DB Dump

```sql
SELECT title, description FROM projects;
-- title: "aGVsbG8gd29ybGQ=..."  (ciphertext)
-- description: "dXNlIHRoaXMgZm9y..."  (ciphertext)

SELECT title, status FROM tasks;
-- title: "encrypted_blob"  (ciphertext)
-- status: "draft"  (readable but meaningless without context)
```

Structural metadata (statuses, timestamps, row counts) is visible. No project names, descriptions, plans, conversations, or acceptance criteria are readable.
